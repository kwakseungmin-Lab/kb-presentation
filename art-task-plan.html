Atomflow 구현

n8n 프로토타입 검증 → YAML 파이프라인 전환

  • Pixel Atomflow 5종 (캐릭터, 프롭, 환경 등)
  • 게임별 커스텀 Atomflow (Jump King, Chess 등)
  • 배경 제거 자동화 (Edge Feather)
  • 프롬프트 빌더 최적화

디펜던시 구조

에셋 간 참조 관계를 정의하여 생성 순서 보장

  • 기본 이미지 에셋 → 애니메이션 참조
  • 캐릭터 베이스 → 표정/동작 변형
  • 환경 배경 → 오브젝트 배치
  • Task 실행 순서 자동 관리
주요 전환: Asset Kind(캐릭터/프롭/환경) 기준 → 제작 방식(Atomflow별) 기준으로 전면 재설계. Prop vs Environment 경계 모호 → 1.5주 소요

Breakout — AI-First 접근

처음부터 AI가 알아서

사람의 직접 개입을 지양하고, AI 중심으로 산출·평가·개선이 가능한 루프를 만든다

Asteroid 방식

사람이 프리셋 설계 → AI가 실행

  • 사람이 프리셋 정의
  • 사람이 디펜던시 설계
  • 사람이 Atomflow 구성
  • AI는 정해진 파이프라인 실행만
Breakout 방식

AI가 Task Plan 자체를 생성

  • AI가 GDD 분석
  • AI가 Task Plan 구조 설계
  • AI가 명세 품질 평가
  • AI가 스스로 프롬프트 개선

Breakout — 컨셉 진화

v2 불변성 → v9 레지스트리 검증까지

각 컨셉은 이전 컨셉의 한계를 보완하며 점진적으로 추가

v2 불변성

Invariance

동일 Task Plan → 어떤 AI든 동일 결과. 형용사 금지, Hex 코드, px 단위

v3 구조성

Structurality

불변성이 벽돌 품질이면, 구조성은 설계도. 계층 구조 + 디펜던시 + 관계 명시 + 맥락 포함

v4 완결성

Completeness

프로젝트 내 모든 에셋이 동일한 스타일 규칙을 따르는 상태

v5 참조 무결성

Reference Integrity

에셋 간 참조가 정확하고 깨지지 않는 상태

v6 출력 계약

Output Contract

출력 형식과 구조가 안정적으로 유지되는 상태

v7~v9

증거 추적 → 레지스트리

Evidence Traceability, Dependency Graph, Registry Validation

핵심 원칙 (불변성): "레퍼런스 이미지가 없어도 텍스트만으로 AI가 상상할 틈이 없으면, 그것이 불변성이다"
