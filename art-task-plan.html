정의 WHAT IS ART TASK PLAN?

Art Task Plan이란?

게임의 GDD(Game Design Document)를 받아서, 그 안에 필요한 모든 아트 에셋을 AI가 생성할 수 있도록 만든 구체적인 작업 명세서

사람 아티스트에게 주는 작업 지시서가 아니라,
AI가 읽고 바로 에셋을 생성할 수 있는 수준의 명세서

핵심 문제: "수박 캐릭터를 그려줘"라고 하면 AI는 매번 다른 결과를 낸다. 어떻게 하면 항상 같은 결과를 내게 할 수 있을까?

ART TASK PLAN이 다루는 것

에셋별 시각 명세

색상, 크기, 스타일, 형태를 숫자로 정의

에셋 간 관계

캐릭터 idle → walk → jump 생성 순서와 의존성

프로젝트 일관성

모든 에셋이 같은 아트 스타일/색상 체계를 따름

실행 완결성

AI가 추가 질문 없이 바로 생성할 수 있는 정보 완비

Asteroid 산출물 WHAT WE BUILT

프리셋 + 스키마 + Task Plan JSON

프리셋으로 시각 속성을 숫자화하고, 스키마로 Task Plan JSON의 전체 구조를 정의하고, 최종 Task Plan JSON을 출력

① 공통 프리셋 (5개) — 수치화, 모든 에셋 공유
BASIC 기본

시각 복잡도, 시점, 구조 구성

FORM 형태

형태, 비율, 대칭, 밀도, 여백

COLOR 색상

색상 수, 채도, 밝기, 대비, 색온도

TEXTURE 질감

디테일, 거칠기, 패턴, 단단함, 노후도

ART STYLE 아트 스타일

사실↔추상, 렌더링, 레트로 시대, 셰이딩, 외곽선 두께

② 종류별 프리셋 — 에셋 타입마다 추가 수치
CHARACTER 캐릭터

표면 유형, 신체 변형, 장비, 부속

ENV/PROP 환경/오브젝트

재질, 형태, 부속

UX/UI 인터페이스

컴포넌트 타입, 시각 스타일, 테두리

VFX 이펙트

파티클 형태, 이펙트 스타일

③ 스키마 — Task Plan JSON의 구조 정의서

Task Plan JSON에 어떤 필드가 있어야 하는지 전체 구조를 정의한 템플릿
프리셋 필드(수치) + 서술형 필드(color_palette, visual_elements, reference_images 등) + 메타 정보(task_id, dependencies, export 설정) 모두 포함

④ Task Plan JSON — 최종 출력물

스키마 구조에 따라 프리셋 값 등을 채운 최종 명세서. AI가 읽고 바로 에셋을 생성할 수 있는 JSON 문서

프리셋 변환 예시
// "매우 밝고 따뜻한 느낌" →
brightness: 8 // 밝기 HSV 80~90%
color_temperature: 8 // 색온도 80~90% 따뜻함
saturation: 7 // 채도 HSV 65~75%
정리

프리셋 = 시각 속성을 숫자화한 값 (1~10)
스키마 = Task Plan JSON의 전체 필드 구조 정의
Task Plan JSON = 스키마 구조에 값을 채운 최종 명세서

접근 방식 APPROACH

세 단계로 진행

STEP 1

먼저 AI로 돌려보기

아무 컨셉 없이 GDD만 주고 Task Plan을 생성시켜봄

→ 추상적, 매번 다른 결과, 정보 부족 등 문제 발견

STEP 2

개념 누적으로 Task Plan 만들기

불변성 → 구조성 → 완결성 순서로 개념을 누적 적용하면서 품질 개선

→ 9개 버전까지 진화

STEP 3

KB를 활용한 메타이터레이션

이터레이션 결과를 KB에 축적하고, 다음 이터레이션의 전략 선택에 재사용

→ 기반 부족으로 KB 기초부터 재학습 중

v2 불변성

INVARIANCE

동일 Task Plan → 어떤 AI든 동일 결과. 형용사 금지, Hex 코드, px 단위

v3 구조성

STRUCTURALITY

불변성이 벽돌 품질이면, 구조성은 설계도. 계층 구조 + 디펜던시 + 관계 명시 + 맥락

v4 완결성

COMPLETENESS

실행에 필요한 모든 정보가 빠짐없이 포함. AI가 추가 질문 없이 바로 실행 가능

v5 참조 무결성

REFERENCE INTEGRITY

에셋 간 참조가 정확하고 깨지지 않는 상태

v6 출력 계약

OUTPUT CONTRACT

출력 형식과 구조가 안정적으로 유지

v7~v9

EVIDENCE → REGISTRY

증거 추적, 디펜던시 그래프, 레지스트리 검증

"불변성(What: 또렷한가) → 구조성(How: 체계적인가) → 완결성(All: 빠진 게 없는가)"

핵심 교훈 KEY LESSONS

두 마일스톤에서 배운 것

Task Plan은 최대한 구체적이어야 한다

AI가 읽고 바로 실행할 수 있을 정도로 구체적이어야 한다. 추상적이면 매번 다른 결과가 나온다. 프리셋으로 숫자화한 이유가 이것.

실험/파이프라인 설계에 시간을 더 쓰자

빨리 돌리는 것보다 실험을 어떻게 설계할지, 파이프라인을 어떻게 구성할지 고민하는 시간이 더 중요했다.

고정 파이프라인의 한계

Asteroid에서 모든 Task를 Atomflow로 해결하려 했지만 한계. 고정된 파이프라인 대신 유연한 Agentic 방향이 필요했다.

기반을 갖추고 설계하라

KB든 어떤 도구든, 동작 원리를 모르고 설계하면 깨진다. 먼저 써보고, 실험해보고, 이해한 다음에 설계해야 한다.
