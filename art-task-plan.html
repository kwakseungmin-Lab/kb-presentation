정의 WHAT IS ART TASK PLAN?

Art Task Plan이란?

게임의 GDD(Game Design Document)를 받아서, 그 안에 필요한 모든 아트 에셋을 AI가 생성할 수 있도록 만든 구체적인 작업 명세서

사람 아티스트에게 주는 작업 지시서가 아니라,
AI가 읽고 바로 에셋을 생성할 수 있는 수준의 명세서

핵심 문제: "수박 캐릭터를 그려줘"라고 하면 AI는 매번 다른 결과를 낸다. 어떻게 하면 항상 같은 결과를 내게 할 수 있을까?

ART TASK PLAN이 다루는 것

에셋별 시각 명세

색상, 크기, 스타일, 형태를 숫자로 정의

에셋 간 관계

캐릭터 idle → walk → jump 생성 순서와 의존성

프로젝트 일관성

모든 에셋이 같은 아트 스타일/색상 체계를 따름

실행 완결성

AI가 추가 질문 없이 바로 생성할 수 있는 정보 완비

프리셋이란? WHAT IS A PRESET?

에셋의 모든 시각 속성을 숫자로 고정하는 템플릿

프리셋은 에셋 하나를 만들기 위해 필요한 모든 시각 속성을 미리 정의해둔 템플릿입니다. 텍스트 설명 대신 숫자/코드만 들어가서, Task Plan에 이 프리셋 값을 넣으면 AI가 매번 동일한 에셋을 생성합니다.

프리셋이 정의하는 항목들
// 프리셋 = 이 모든 것을 숫자로 고정
art_style: "flat_design" // 아트 스타일
viewpoint: "front, eye-level, 0°" // 시점
canvas: 64×64 px // 캔버스 크기
form: "원형, 80% coverage" // 형태
color_primary: #2E8B57 // 주 색상
color_accent: #FF6B6B // 보조 색상
shadow: 2단계, 좌상단 광원, 20%
outline: none // 없는 것도 명시
seed: 42 // 재현 파라미터
프리셋 없이 (텍스트 명세)
"밝은 파란색 수박 캐릭터를 그려주세요"
"적당한 크기로 귀여운 느낌으로"
// → AI가 매번 다른 결과
프리셋 적용 (숫자 명세)
canvas: 64×64, color: #2E8B57,
shadow: 2단계, outline: none
// → 어떤 AI든 동일한 결과
핵심 원칙 3가지

형용사 금지

"밝은" → #87CEEB

숫자로 고정

"적당히" → 64px

없음도 명시

outline: none

접근 방식 APPROACH

세 단계로 진행

STEP 1

먼저 AI로 돌려보기

아무 컨셉 없이 GDD만 주고 Task Plan을 생성시켜봄

→ 추상적, 매번 다른 결과, 정보 부족 등 문제 발견

STEP 2

개념 누적으로 Task Plan 만들기

불변성 → 구조성 → 완결성 순서로 개념을 누적 적용하면서 품질 개선

→ 9개 버전까지 진화

STEP 3

KB를 활용한 메타이터레이션

이터레이션 결과를 KB에 축적하고, 다음 이터레이션의 전략 선택에 재사용

→ 기반 부족으로 KB 기초부터 재학습 중

v2 불변성

INVARIANCE

동일 Task Plan → 어떤 AI든 동일 결과. 형용사 금지, Hex 코드, px 단위

v3 구조성

STRUCTURALITY

불변성이 벽돌 품질이면, 구조성은 설계도. 계층 구조 + 디펜던시 + 관계 명시 + 맥락

v4 완결성

COMPLETENESS

실행에 필요한 모든 정보가 빠짐없이 포함. AI가 추가 질문 없이 바로 실행 가능

v5 참조 무결성

REFERENCE INTEGRITY

에셋 간 참조가 정확하고 깨지지 않는 상태

v6 출력 계약

OUTPUT CONTRACT

출력 형식과 구조가 안정적으로 유지

v7~v9

EVIDENCE → REGISTRY

증거 추적, 디펜던시 그래프, 레지스트리 검증

"불변성(What: 또렷한가) → 구조성(How: 체계적인가) → 완결성(All: 빠진 게 없는가)"

KB 메타이터레이션 WHY KB FOR META ITERATION?

KB로 하려던 것: 다음 전략을 과거로부터 고르기

이터레이션을 돌릴 때마다 결과가 쌓이는데, 그 결과를 구조적으로 기억시켜서 다음 이터레이션의 전략을 자동으로 선택하게 하고 싶었다

KB에 기대한 역할

과거 이터레이션 기억

iter_007의 점수, 실패 패턴, 적용했던 개념, 전략 변경 이력을 엔티티/릴레이션으로 축적

다음 전략 자동 선택

축적된 패턴을 retrieve해서 "이번엔 어떤 프롬프트/개념/검증 기준으로 돌릴지" AI가 스스로 결정

개념(Concept) 자동 진화

"불변성만으로 부족했다" → "구조성을 추가하자"를 AI가 KB에서 근거를 찾아 판단

기대했던 흐름
이터레이션 실행 → 결과/점수/실패 패턴 발생
↓
KB에 구조적으로 저장 (엔티티 + 릴레이션)
↓
다음 이터레이션 시 KB에서 retrieve
↓
전략 자동 선택 → 더 나은 Task Plan 생성

현실: KB의 retrieve, 엔티티, 릴레이션을 제대로 이해하지 못한 채 설계하려다 깨짐. 지금 KB 기초부터 다시 습득 중.

도메인 지식이 먼저다

체계적으로 축적되지 않으면 프롬프트 미세 조정만 반복. KB든 문서든, 도메인 이해가 AI와 사람 모두에게 필수.

잘못된 것을 최적화하지 마라

먼저 작동하게 만들고 → 문제를 즉시 해결 → 도메인 지식을 한 문서에 → 사람 개발자가 평가 루프에.

스코프를 제어하라

목적, 수행 내용, 검증 방법, 기대 효과 — 4가지를 먼저 정리해야 본능적 리팩토링을 제어 가능.

기반을 갖추고 설계하라

KB를 써보지도 않고 KB 기반 파이프라인을 설계하면 깨진다. retrieve, top-k, 실제를 먼저.
