접근 방식 APPROACH

먼저 해보고, 문제를 찾고, 컨셉으로 고치기

AI한테 먼저 Task Plan을 만들게 해보고, 발견된 문제를 컨셉으로 하나씩 잡아나가는 방식

STEP 1

AI에게 먼저 시켜봄

아무 컨셉 없이 GDD만 주고 Task Plan 생성

→ 추상적, 매번 다른 결과, 정보 부족

STEP 2

문제별 컨셉 추가

불변성 → 구조성 → 완결성 순서로 누적 적용

→ 발견된 문제에 맞는 컨셉을 추가

STEP 3

자동 평가 & 개선 루프

Evaluator로 평가 → Evolver가 프롬프트 수정

→ 재생성하며 품질 반복 개선

v2 불변성

v3 구조성

STRUCTURALITY

불변성이 벽돌 품질이면, 구조성은 설계도. 계층 구조 + 디펜던시 + 관계 명시 + 맥락

v4 완결성

COMPLETENESS

실행에 필요한 모든 정보가 빠짐없이 포함. AI가 추가 질문 없이 바로 실행 가능

v5 참조 무결성

REFERENCE INTEGRITY

에셋 간 참조가 정확하고 깨지지 않는 상태

v6 출력 계약

OUTPUT CONTRACT

출력 형식과 구조가 안정적으로 유지

v7~v9

EVIDENCE → REGISTRY

증거 추적, 디펜던시 그래프, 레지스트리 검증

"불변성(What: 또렷한가) → 구조성(How: 체계적인가) → 완결성(All: 빠진 게 없는가)"

두 마일스톤에서 배운 것

도메인 지식이 먼저다

체계적으로 축적되지 않으면 프롬프트 미세 조정만 반복. KB든 문서든, 도메인 이해가 AI와 사람 모두에게 필수.

잘못된 것을 최적화하지 마라

먼저 작동하게 만들고 → 문제를 즉시 해결 → 도메인 지식을 한 문서에 → 사람 개발자가 평가 루프에.

스코프를 제어하라

목적, 수행 내용, 검증 방법, 기대 효과 — 4가지를 먼저 정리해야 본능적 리팩토링을 제어 가능.

기반을 갖추고 설계하라

KB를 써보지도 않고 KB 기반 파이프라인을 설계하면 깨진다. retrieve, top-k, 실제를 먼저.
