정의 WHAT IS ART TASK PLAN

Art Task Plan이란?

게임의 GDD(Game Design Document)를 받아서, 그 안에 필요한 모든 아트 에셋을 AI가 생성할 수 있도록 만든 구체적인 작업 명세서

사람 아티스트에게 주는 작업 지시서가 아니라,
AI가 읽고 바로 에셋을 생성할 수 있는 수준의 명세서

핵심 문제: "수박 캐릭터를 그려줘"라고 하면 AI는 매번 다른 결과를 낸다. 어떻게 하면 항상 같은 결과를 내게 할 수 있을까?

ART TASK PLAN이 다루는 것

프리셋 시스템과 Task Plan JSON

시각 속성을 수치화하여 AI가 일관된 결과를 생성하도록 하는 시스템 구축

① 공통 프리셋 — 모든 에셋이 공유하는 기본 시스템
Common_Presets/

2D 공통:
• Color_presets.json - 색상 시스템
• saturation (채도), brightness (밝기): 1-10 → HSV 0-100%
• contrast (대비), color_temperature (색온도) 수치화
3D 공통:
• 3D_Presets/material_presets.json
• PBR 재질 속성: roughness (거칠기) 1-10
• reflectivity (반사도), metalness (금속성) 수치화

② 종류별 프리셋 — 에셋 타입마다 특화된 속성
PROP 오브젝트

basic (크기, 복잡도), form (형태), texture (재질), artstyle

ENV 환경

Biomes (숲, 사막, 설원), Material_Properties (잔디, 흙, 눈), Weather, Time_of_Day

CHARACTER 캐릭터

Humanoid, Animal, Amorphous별 body_parts, face, clothing

VFX/UI 이펙트/UI

Particle_Shape, Trail_Motion, Glow_Bloom, UI schemas

③ 스키마 — Task Plan JSON의 구조 정의서

Task Plan JSON에 어떤 필드가 있어야 하는지 전체 구조를 정의한 템플릿
프리셋 필드(수치) + 서술형 필드(palette, context) + 메타 정보(task_id, dependencies) 모두 포함

④ Task Plan JSON — 최종 출력물

스키마 구조에 따라 프리셋 값을 채운 최종 명세서. AI가 읽고 바로 에셋을 생성할 수 있는 JSON 문서

실제 프리셋 예시 (Environment - Biomes)
"forest": {
  vegetation_density: 7 // 식물 밀도
  tree_type: ["oak", "pine"]
  saturation: 6 // 공통 프리셋 참조
}
핵심 차이점

공통 = 색상 시스템 (모든 에셋 공유)
종류별 = 에셋 타입별 완전히 다른 체계
목적 = 추상적 표현 → 정확한 수치 변환

MILESTONE 02

Breakout

AI로 돌려보기 → 개념 누적 → KB 메타이터레이션

접근 방식 APPROACH

세 단계로 진행

STEP 1

먼저 AI로 돌려보기

아무 컨셉 없이 GDD만 주고 Task Plan을 생성시켜봄

→ 추상적, 매번 다른 결과, 정보 부족 등 문제 발견

STEP 2

개념 누적으로 Task Plan 만들기

불변성 → 구조성 → 완결성 순서로 개념을 누적 적용하면서 품질 개선

→ 9개 버전까지 진화

STEP 3

KB를 활용한 메타이터레이션

이터레이션 결과를 KB에 축적하고, 다음 이터레이션의 전략 선택에 재사용

→ 기반 부족으로 KB 기초부터 재학습 중

컨셉 진화 CONCEPT EVOLUTION

v2 불변성 → v8까지

각 컨셉은 이전 이터레이션의 문제를 보완하며 누적 적용

v2 불변성

INVARIANCE

동일 Task Plan → 어떤 AI든 동일 결과. 형용사 금지, Hex 코드, px 단위

v3 구조성

STRUCTURALITY

불변성이 벽돌 품질이면, 구조성은 설계도. 계층 구조 + 디펜던시 + 관계 명시 + 맥락

v4 완결성

COMPLETENESS

실행에 필요한 모든 정보가 빠짐없이 포함. AI가 추가 질문 없이 바로 실행 가능

v5 참조 무결성

REFERENCE INTEGRITY

각 Task JSON이 다른 파일 참조 없이 독립 제작 가능. variant_of 관계도 명확히

v6 출력 계약

OUTPUT CONTRACT

JSON 구조/필드명/파일명이 항상 동일. schema_version으로 버전 관리

v7 증거 추적성

EVIDENCE TRACEABILITY

생성 과정과 결정 근거를 추적 가능하게 기록

v8 디펜던시 그래프

DEPENDENCY GRAPH

의존관계 검증, 순환참조 방지, 실행순서 정합성

"불변성(What: 또렷한가) → 구조성(How: 체계적인가) → 완결성(All: 빠진 게 없는가)"

핵심 교훈 KEY LESSONS

두 마일스톤에서 배운 것

Task Plan은 최대한 구체적이어야 한다

AI가 읽고 바로 실행할 수 있을 정도로 구체적이어야 한다. 추상적이면 매번 다른 결과가 나온다. 프리셋으로 숫자화한 이유가 이것.

실험/파이프라인 설계에 시간을 더 쓰자

빨리 돌리는 것보다 실험을 어떻게 설계할지, 파이프라인을 어떻게 구성할지 고민하는 시간이 더 중요했다.

고정 파이프라인의 한계

Asteroid에서 모든 Task를 Atomflow로 해결하려 했지만 한계. 고정된 파이프라인 대신 유연한 Agentic 방향이 필요했다.

기반을 갖추고 설계하라

KB든 어떤 도구든, 동작 원리를 모르고 설계하면 깨진다. 먼저 써보고, 실험해보고, 이해한 다음에 설계해야 한다.
