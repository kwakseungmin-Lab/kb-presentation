01 — 개념과 정의

KB란 무엇인가?

관계, 상태, 이력을 구조적으로 저장해서 다음 실행에서 다시 쓰기 위한 지식 레이어

Document Retrieval KB

문서 기반

문서를 통째로 저장 → 비슷한 문서를 찾아오는 방식

  • 사람이 읽기 자연스러움
  • 원문 보관에 적합
  • 설계 비용 낮음
Graph / Fact-Memory KB

현재 KB 방식

사실(fact)을 entity/relation으로 쪼개서 저장하고 연결

  • 구조적 재사용에 강함
  • 관계 추적 가능
  • 다음 실행 판단에 활용
핵심 구분: 사람이 읽을 것은 문서에 두고, 시스템이 다시 찾아 써야 하는 것은 KB에 둔다.
02 — 권장 구조

문서 레이어 + KB 레이어

두 레이어를 명확히 분리하는 것이 가장 자연스러운 구조

문서/Wiki 레이어

  • 사람이 읽는 설명
  • 배경 맥락
  • 최종 정리 / 보고서
  • Raw document, note, evidence

KB 레이어

  • 실행에 필요한 구조화 정보
  • Version 관계
  • 교훈, 성과
  • 다음 실행에서 재사용할 연결 정보
주의: KB가 문서를 대체하는 것이 아닙니다. 원문/보고서/최종 근거가 목적이면 문서가 낫고, 다음 실행 판단에 다시 써야 하는 기억이 목적이면 KB가 낫습니다.
03 — 엔티티

Entity: KB의 기본 단위

KB 안에서 독립적으로 이름 붙일 수 있는 대상

// 엔티티 필드 구조
{
  entity_id: "goal_spec::suika::v1" // 시스템 식별 ID
  label: "GoalSpec" // 사람이 읽는 이름/타입
  summary: "suika 프로젝트 목표 명세 v1" // 짧은 설명
  properties: { ... } // 부가 메타데이터 (내용은 엔티티별로 다름)
  updated_at: "2026-04-15T..."
}
ID 규칙: entity_type::project::version → 보면 바로 뭔지 알 수 있고, 중복 없음
필드역할필수
entity_id시스템에서 정확히 식별하는 IDO
label사람이 읽는 이름/타입O
summary짧은 설명
properties부가 메타데이터 (내용은 엔티티별로 다름, 비어있을 수도 있음)
04 — 릴레이션

Relation: 구조적 기억의 핵심

두 엔티티 사이의 방향성 있는 의미 연결

retrieve_kb_graph
retrieves_from_module
predicate
knowledge_base_module

"retrieve_kb_graph가 knowledge_base_module로부터 retrieval을 수행한다"

{
  relation_id: "rel_001",
  source_entity_id: "azle_tool_retrieve_kb_graph",
  target_entity_id: "azle_knowledge_base_module",
  predicate: "retrieves_from_module" // 관계의 본질
}
물리 엣지 타입은 하나: KB_RELATES — 실제 의미 구분은 predicate에 들어간다. 방향을 항상 고정할 것!

Predicate 예시

Predicate의미

05 — 검색 vs 조회

검색과 조회는 다르다

검색 = 질의를 넣고 관련 대상을 찾아내는 것  /  조회 = 이미 저장된 걸 읽고 확인하는 것

검색 (Retrieval)

retrieve_kb_graph

의미 기반 semantic retrieval

  • embedding 기반 유사도 검색
  • top-k ranking으로 후보 선별
  • hop expansion으로 그래프 펼침
  • query → 관련 대상을 찾아냄
exact entity_id lookup이 보장되지 않음
조회 (Read / Lookup)

4가지 방식

이미 저장된 것을 읽고 확인

browse_kb_entities
literal/filter 조회 — 존재 확인, 목록 훑기
snapshot
전체 상태 inspection — score/ranking 없음
get_kb_entity_neighbors
특정 엔티티 주변 그래프 조회
get_kb_stats
엔티티/릴레이션 수 통계
retrieve는 "뜻으로 찾기", browse는 "보이는 값으로 훑기" — 목적에 맞는 방식을 선택
05-B — Semantic Retrieval 상세

retrieve_kb_graph 동작 흐름

① Query 입력
→
② 임베딩 변환
→
③ 유사 엔티티 점수 매김
→
④ top-k 선택
→
⑤ hop_depth 그래프 펼침

잘 맞는 경우

  • 개념/패턴 찾기
  • 관련 문맥 찾기
  • 기억 회수
  • "이 주제와 가까운 게 뭐지?"

약한 경우

  • Exact literal match
  • 정확한 entity_id 검증
  • Keyword audit

정확한 텍스트를 입력해도 유사도 검색을 해버릴 수 있다

entity_id를 정확하게 찾는 것이 보장되지 않음 — exact lookup이 필요하면 browse를 사용
06 — top-k와 hop_depth

top-k와 hops의 차이

top-k

초기 semantic match 후보를 몇 개까지 남길지

// 100개 엔티티가 점수순 정렬됨
top_k = 5
// → 상위 5개만 matches로 채택
top_k는 matches 리스트만 제한.
최종 nodes/relations 총량을 제한하는 값이 아님

hop_depth

seed match 주변 관계를 몇 단계까지 펼칠지

hop_depth = 0 // seed만
hop_depth = 1 // 직접 이웃까지
hop_depth = 2 // 이웃의 이웃까지
키우면 문맥이 풍부해지지만
노이즈 증가, 그래프 비대화 주의

예시: top_k=5, hop_depth=1

5

matches

12

nodes

9

relations

중복은 제거되어 반환됨

09 — 청킹 (Chunking)

문서는 어떻게 쪼개지는가?

AST section-aware markdown chunking — 구조를 최대한 보존하면서 분할

Frontmatter 분리
→
Heading 기준 Section
→
Block 단위 분할
→
Oversize 추가 분할

Block 단위

  • paragraph (문단)
  • list (목록)
  • blockquote (인용)
  • fenced code block

Chunk 메타데이터

heading_path: "GoalSpec > 평가기준"
section_order: 3
chunk_index: 5
file_path: "docs/goal_spec.v1.md"
content_sha256: "a1b2c3..."

통계 / 검증

  • max chunk: ~1612 chars
  • 10,000자 초과: 0
  • chunk ≤ 1700 chars
  • 큰 chunk: table, list, JSON
무작정 글자 수로 자르는 게 아니라, heading/section/block 구조를 최대한 보존하면서 자름

11 — 왜 KB를 쓰는가

KB를 쓰는 진짜 이유

반복 실행 사이에서 재사용해야 하는 관계·상태·이력을 구조적으로 기억시켜 다음 의사결정에 쓰기 위해

🔄

반복 실행 간 기억

과거 iteration 결과, 점수, 실패 패턴, 전략 이력을 축적

🎯

전략 선택 근거

다음 iteration 전략에 재사용: prompt, evolver mode, 검증 기준

📊

추세와 패턴 추적

실패 패턴, 품질 추세, 전략 이력을 보고 판단

문서가 나은 경우
  • 사람이 읽고 이해해야 하는 설명
  • 공유용 문서, 최종 근거/보고서
  • 단순 보관/검색이 주목적
KB가 나은 경우
  • 버전 ↔ 정책/실행 결과 연결
  • trial/evaluation이 다음 실행에 재사용
  • 다음 의사결정에 써야 하는 운영 지식
