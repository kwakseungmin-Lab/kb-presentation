01 — KB란? WHAT IS KB?

관계·상태·이력을 구조적으로 저장하는 지식 레이어

현재 KB는 document retrieval이 아닌 graph/fact-memory 형태 — 사실을 entity/relation으로 쪼개서 저장하고 연결

문서가 맞는 경우
  • 사람이 읽고 이해해야 하는 설명
  • 배경 맥락, 최종 정리, 보고서
  • 원문 보관/공유가 주목적
KB가 맞는 경우
  • 실행에 필요한 구조화 정보
  • Version 관계, 교훈, 성과
  • 다음 실행에서 재사용할 연결 정보
GRAPH KB의 특징
  • 사실(fact)을 엔티티로 저장
  • 엔티티 간 연결을 릴레이션으로 표현
  • 의미 기반 검색(retrieve)으로 관련 정보 찾기
  • 연결을 따라가며 문맥 확장(hop)
핵심: 사람이 읽을 것은 문서에, 시스템이 다시 찾아 써야 하는 것은 KB에 둔다.

04 — 릴레이션 RELATION

Relation: 구조적 기억의 핵심

두 엔티티 사이의 방향성 있는 의미 연결

{
  relation_id: "rel_001",
  source_entity_id: "azle_tool_retrieve_kb_graph",
  target_entity_id: "azle_knowledge_base_module",
  predicate: "retrieves_from_module"
}

물리 엣지 타입은 하나: KB_RELATES — 의미 구분은 predicate에. 방향 항상 고정!

PREDICATE 예시
Predicate의미
refers_to이 entity가 저 entity를 참고
supersedes이 버전이 이전 버전을 대체
evaluated_by이 Task Plan이 이 평가를 받음
caused_by이 실패가 이 원인으로 발생
uses_tool이 entity가 저 tool을 사용
"KB가 검색 인덱스가 아니라 구조적 기억이 되게 만드는 핵심이 릴레이션"
SECTION

검색 vs 조회

검색 = 질의로 찾아내는 것  /  조회 = 이미 저장된 걸 읽는 것

05 — 검색 vs 조회 RETRIEVAL VS READ

검색과 조회는 다르다

검색 (RETRIEVAL)

retrieve_kb_graph

의미 기반 semantic retrieval

  • embedding 기반 유사도 검색
  • top-k ranking으로 후보 선별
  • hop expansion으로 그래프 펼침
  • query → 관련 대상을 찾아냄
exact entity_id lookup이 보장되지 않음
조회 (READ / LOOKUP)

4가지 방식

이미 저장된 것을 읽고 확인

browse_kb_entities
literal/filter 조회 — 존재 확인, 목록 훑기
snapshot
전체 상태 inspection — score/ranking 없음
get_kb_entity_neighbors
특정 엔티티 주변 그래프 조회
get_kb_stats
엔티티/릴레이션 수 통계
retrieve는 "뜻으로 찾기", browse는 "보이는 값으로 훑기" — 목적에 맞는 방식을 선택

06 — top-k와 hop_depth

top-k와 hops의 차이

top-k

초기 semantic match 후보를 몇 개까지 남길지

// 100개 엔티티가 점수순 정렬됨
top_k = 5
// → 상위 5개만 matches로 채택
top_k는 matches 리스트만 제한. 최종 nodes/relations 총량은 아님
hop_depth

seed match 주변 관계를 몇 단계까지 펼칠지

hop_depth = 0 // seed만
hop_depth = 1 // 직접 이웃까지
hop_depth = 2 // 이웃의 이웃까지
키우면 문맥 풍부 but 노이즈 증가, 그래프 비대화 주의

10 — 왜 KB를 쓰는가 WHY KB?

KB를 쓰는 진짜 이유

반복 실행 사이에서 재사용해야 하는 관계·상태·이력을 구조적으로 기억시켜 다음 의사결정에 쓰기 위해

문서가 나은 경우
  • 사람이 읽고 이해해야 하는 설명
  • 공유용 문서, 최종 근거/보고서
  • 단순 보관/검색이 주목적
KB가 나은 경우
  • 버전 ↔ 정책/실행 결과 연결
  • trial/evaluation이 다음 실행에 재사용
  • 다음 의사결정에 써야 하는 운영 지식

11 — 설계 원칙 DESIGN PRINCIPLES

KB 설계 시 꼭 먼저 정할 것

1. 엔티티를 어떻게 나눌지

어떤 대상을 독립 엔티티로 관리할지 단위를 정함

2. RELATION 종류

refers_to, supersedes, evaluated_by, caused_by 등

3. ID 규칙

중복 없이 exact lookup 가능해야 함

4. 경계 설정

운영 메모, 설계, 정책, 상태가 섞이지 않게

잘못 설계하면 문서보다 더 나빠진다: 중복 엔티티, 애매한 relation, 불명확한 predicate, 경계 모호함
