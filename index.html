01 — 개념과 정의 DEFINITION

KB란 무엇인가?

관계, 상태, 이력을 구조적으로 저장해서 다음 실행에서 다시 쓰기 위한 지식 레이어

문서가 맞는 경우
  • 사람이 읽고 이해해야 하는 설명
  • 배경 맥락, 최종 정리, 보고서
  • 원문 보관/공유가 주목적
KB가 맞는 경우
  • 단순 저장/불러오기가 아니라 관계, 상태, 이력 같은 구조가 중요할 때
  • 쌓은 지식들이 다음 의사결정에 다시 써야 하는 운영 지식일 때
  • 여러 에이전트나 시스템이 같은 구조 지식을 필요로 할 때
Azle KB = GRAPH / FACT-MEMORY 형태

사실(fact)을 entity와 relation으로 쪼개서 저장하고 연결하는 구조

  • 구조적 재사용에 강함
  • 관계 추적 가능
  • 다음 실행 판단에 활용
핵심: 사람이 읽을 것은 문서에 두고, 시스템이 다시 찾아 써야 하는 것은 KB에 둔다.

03 — 엔티티 ENTITY

Entity: KB의 기본 단위

KB 안에서 독립적으로 이름 붙일 수 있는 대상

// 엔티티 필드 구조
{
  entity_id: "goal_spec::suika::v1"
  label: "GoalSpec"
  summary: "suika 프로젝트 목표 명세 v1"
  properties: { ... }
  updated_at: "2026-04-15T..."
}
ID 규칙: entity_type::project::version → 보면 바로 뭔지 알 수 있고, 중복 없음
필드역할필수
entity_id시스템에서 정확히 식별하는 IDO
label사람이 읽는 이름/타입O
summary짧은 설명
properties부가 메타데이터 (타입별로 필요한 정보)
properties 필드: 이름은 공통이지만 안의 내용은 엔티티마다 다를 수 있고, 아예 비어 있을 수도 있다.
04 — 릴레이션 RELATION

Relation: 구조적 기억의 핵심

두 엔티티 사이의 방향성 있는 의미 연결

// API/조회 모델 기준
{
  relation_id: "rel_001",
  source_entity_id: "azle_tool_retrieve_kb_graph",
  target_entity_id: "azle_knowledge_base_module",
  predicate: "retrieves_from_module",
  description: "retrieve_kb_graph가 KB 모듈에서 retrieval 수행",
  properties: { ... },
  updated_at: "2026-04-15T..."
}

Azle KB에서 물리 엣지 타입은 하나: KB_RELATES — 의미 구분은 predicate에 둔다

Outgoing / Incoming = 엔티티 관점에서 본 릴레이션 방향
A → uses_tool → B 일 때: A의 Outgoing +1, B의 Incoming +1

릴레이션 구조 (API/조회 모델)
필드역할필수
relation_id릴레이션 고유 식별자O
source_entity_id관계의 출발 엔티티 (이 엔티티 기준 Outgoing)O
target_entity_id관계의 도착 엔티티 (이 엔티티 기준 Incoming)O
predicate관계의 의미 (uses_tool, supersedes 등)O
description사람이 읽는 짧은 설명
properties부가 메타데이터 (저장 시 properties_json)
updated_at마지막 수정 시각
구조 요약: source → predicate → target 방향의 의미 연결. predicate가 관계의 본질, description/properties는 보조. 저장 시엔 KB_RELATES 단일 엣지로 저장되며, source와 target은 관계의 양 끝 엔티티입니다.
04-B — 릴레이션 설계 원칙 RELATION DESIGN

왜 릴레이션이 중요한가 + 설계 원칙

릴레이션이 중요한 4가지 이유

맥락 추적

관련된 것들을 함께 볼 수 있음

hop expansion

주변 그래프를 단계적으로 펼침

설명 가능한 구조

관계를 통해 결과 추적 가능

태그보다 선명

방향 + predicate로 정확한 표현

설계 원칙 4가지

1. predicate는 일관된 어휘 — 같은 관계에 다른 단어 쓰면 검색이 깨짐

2. 방향 항상 고정 — tool → api (반대로도 쓰면 혼란)

3. description에 짧은 설명 — predicate만으로 부족한 맥락 보완

4. 관계의 본질은 predicate에, 부가 정보만 properties에

PREDICATE 예시
Predicate의미
refers_to이 entity가 저 entity를 참고
supersedes이 버전이 이전 버전을 대체
evaluated_by이 Task Plan이 이 평가를 받음
caused_by이 실패가 이 원인으로 발생
uses_tool이 entity가 저 tool을 사용
documented_in이 entity의 문서가 저기에 있음
complements_reference이 entity가 저 레퍼런스를 보완
retrieves_from_module이 entity가 저 모듈에서 데이터 가져옴
"KB가 검색 인덱스가 아니라 구조적 기억이 되게 만드는 핵심이 릴레이션"

05 — 검색과 조회 RETRIEVAL VS READ

검색 2가지 + 조회 4가지

검색 1: EXACT / METADATA LOOKUP

정확한 ID나 key를 이미 알고 있을 때

  • 특정 entity_id로 직접 조회
  • label이나 properties 필터링
  • 의미 유사도 검색이 아님
  • "비슷한 것"이 아니라 정확히 찾는 것
이미 알고 있는 ID나 key로 정확한 조회
검색 2: SEMANTIC RETRIEVAL

retrieve_kb_graph

의미 기반 semantic retrieval

  • embedding 기반 유사도 검색
  • top-k ranking으로 후보 선별
  • hop expansion으로 그래프 펼침
  • exact entity_id lookup 보장 안 됨

06 — top-k와 hop_depth

top-k와 hops의 차이

top-k

초기 semantic match 후보를 몇 개까지 남길지

// 100개 엔티티가 점수순 정렬됨
top_k = 3 // 상위 3개만
top_k = 5 // 상위 5개
top_k = 8 // 상위 8개
top_k = 12 // 더 많은 후보 포함
top_k는 matches 리스트만 제한. 최종 nodes/relations 총량은 아님
hop_depth

seed match 주변 관계를 몇 단계까지 펼칠지

hop_depth = 1 // seed + 직접 이웃
hop_depth = 2 // seed + 이웃 + 이웃의 이웃
hop_depth = 3 // 3단계까지 확장
키우면 문맥 풍부 but 노이즈 증가, 그래프 비대화 주의
예시: top_k=5, hop_depth=3 → "유사도 높은 5개를 찾고, 그 5개에서 relation을 따라 3단계까지 주변 그래프를 확장"

09 — Art Task Plan iteration 적합한가?

왜 KB를 쓰나?

KB를 쓰는 이유는 문서를 저장하기 위해서가 아니라, 반복 실행 사이에서 재사용해야 하는
관계·상태·이력을 구조적으로 기억시켜 다음 의사결정에 쓰기 위해서

왜 KB를 쓰나?
  • 문서는 설명과 공유에 강하고, KB는 구조화된 재사용 기억에 강합니다
  • 원문/보고서/최종 근거가 목적이면 문서가 낫고, 다음 실행 현장에 다시 써야 하는 기억이면 KB가 낫다
전략 선택
  • 다음 iteration에서 실행 방식을 어떻게 바꿀지 정하는 것
  • 이터레이션 실행 → 성과/교훈 발생
  • 재사용 가능한 형태로 KB에 저장
  • 다음 이터레이션 시작할 때 KB에서 가져옴
본질
  • 과거 경과, 실패 패턴, 근거를 남기는 히스토리/메모리 계층
  • 조회 가능하고 추적 가능해야 함
  • KB는 최신 진실을 보장하는 장치가 아니라, 기억을 외부화하고 재조회 가능하게 만드는 장치에 가깝다

10 — 설계 원칙 DESIGN PRINCIPLES

KB 설계 시 꼭 먼저 정할 것

1. 엔티티를 어떻게 나눌지

어떤 대상을 독립 엔티티로 관리할지 단위를 정함

2. RELATION 종류

어떤 관계들을 정의할지 (refers_to, supersedes 등)

3. ID 규칙

중복 없이 exact lookup 가능해야 함

4. 경계 설정

운영 메모, 설계, 정책, 상태가 섞이지 않게

잘못 설계하면 문서보다 더 나빠진다: 중복 엔티티, 애매한 relation, 불명확한 predicate, 경계 모호함
